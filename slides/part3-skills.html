Qu'est-ce qu'une Skill ?

Une Skill est un dossier d'instructions, de scripts et de ressources que Claude charge dynamiquement pour améliorer ses performances sur des tâches spécifiques.

Structure d'une Skill 
ma-skill/
├── SKILL.md        ← Requis
├── references/     ← Docs détaillées
│   └── api.md
├── scripts/        ← Scripts exécutables
│   └── validate.sh
├── assets/         ← Ressources
└── evals/          ← Tests de validation
    └── evals.json
Où les placer ?
PROJET .claude/skills/ USER ~/.claude/skills/

💡 Analogie : comme un guide d'onboarding pour un nouvel employé. Pas besoin de lui expliquer le contexte à chaque fois.

La divulgation progressive

Claude ne charge que ce dont il a besoin, quand il en a besoin. La quantité de contexte est effectivement illimitée.

1️⃣
name + description

Toujours en mémoire — Claude décide si la skill est pertinente

2️⃣
Contenu du SKILL.md

Chargé si la skill est jugée pertinente

3️⃣
Fichiers de référence, scripts, assets

Chargés uniquement si nécessaires pour la tâche

Le fichier SKILL.md

Format obligatoire 
---
name: nom-de-la-skill        # (obligatoire)
description: Ce que fait la skill
  et QUAND l'utiliser (crucial !) # (obligatoire)
model: haiku                 # (optionnel) haiku | sonnet | opus
allowed-tools: [Bash, Read]  # (optionnel) restreint les outils disponibles
---

# Nom de la Skill            # (obligatoire)

## Ce que tu dois faire      # (obligatoire)

Instructions détaillées...

## Gotchas (erreurs courantes) # (optionnel)

- Ne jamais faire X car...
- Toujours vérifier Y avant...
⚠️ Le champ description est crucial

C'est ce que Claude lit pour décider d'utiliser la skill. Soyez précis sur les déclencheurs.

🛠 TP 1 — Skill "Code Review" (8 min)

Objectif : créer une skill de revue de code qui enforce vos standards d'équipe.

1. Créer la structure 
mkdir -p .claude/skills/code-review
touch .claude/skills/code-review/SKILL.md
2. Le SKILL.md 
---
name: code-review
description: Effectue une revue de code selon
  nos standards. Utiliser quand l'utilisateur
  demande une "revue" ou "review".
model: haiku
allowed-tools: [Read, Bash]
---

# Code Review — Standards Équipe

## Checklist
🔒 Sécurité — Pas de secrets en dur
📖 Lisibilité — Fonctions < 30 lignes
⚡ Performance — Pas de N+1 queries
🧪 Tests — Cas nominaux couverts

## Format de réponse
Résumé en 1-2 phrases, puis :
### 🔴 Bloquants — problème + numéro de ligne
### 🟡 Améliorations — problème + suggestion
### 🟢 Points positifs — au moins un
3. Tester

async function loginUser(username, password) {
  const user = await db.query(
    `SELECT * FROM users WHERE username = '${username}'`
  );
  if (!user) return null;

  if (user.password === password) {
    const token = Math.random().toString(36);
    await db.query(`UPDATE users SET token='${token}' WHERE id=${user.id}`);
    await db.query(`INSERT INTO logs VALUES (${user.id}, 'login', NOW())`);
    await db.query(`SELECT * FROM roles WHERE user_id=${user.id}`);
    await db.query(`SELECT * FROM perms WHERE user_id=${user.id}`);
    console.log("User logged in: " + username);
    return token;
  }
  return false;
}

🛠 TP 2 — Skill avec fichiers de référence (7 min)

Objectif : créer une skill "commit-message" qui génère des commits conventionnels parfaits.

SKILL.md 
---
name: commit-message
description: Génère des messages de commit
  Conventional Commits. Utiliser quand
  l'utilisateur veut "commiter".
---

## Processus
1. git diff --staged
2. Identifier le type (feat, fix, chore…)
3. Rédiger selon le format

## Format
<type>(<scope>): <description>

[corps si changement complexe]
references/conventional-commits.md

Créer ce fichier et y coller le contenu :

# Conventional Commits — Référence

## Types
- feat     : nouvelle fonctionnalité
- fix      : correction de bug
- chore    : maintenance, dépendances, CI
- refactor : refactoring sans bug fix ni feature
- docs     : documentation uniquement
- test     : ajout ou modification de tests
- style    : formatage, pas de logique

## Format
<type>(<scope>): <description courte>

[corps optionnel si changement complexe]

## Exemples
feat(auth): add OAuth2 login
fix(cart): prevent negative quantity
chore(deps): update jest to v29
docs(readme): add setup instructions

Claude consulte ce fichier uniquement si nécessaire. Progressive Disclosure !
